# MySQL checkpointer — host integration

`MySQLCheckpointer` (aiomysql) is the MySQL counterpart of the Postgres
checkpointer — same model, MySQL-compatible types, and `KEY`
partitioning on `thread_id`. **It never issues DDL at runtime** — your
application owns the schema lifecycle through Alembic, and cubepi only
reads and writes rows. Entering the async context manager
(`async with MySQLCheckpointer(...)`) verifies the schema and raises
`CubepiSchemaUninitialized` / `CubepiSchemaMismatch` if the expected
migrations haven't been applied.

Requires MySQL 8.0.13+ (JSON expression defaults) on InnoDB.

Install with the extra:

```bash
pip install 'cubepi[mysql]'
```

## Using the checkpointer

```python
from cubepi.checkpointer.mysql import MySQLCheckpointer

# Entering the context manager connects the pool and verifies the schema.
async with MySQLCheckpointer("mysql://user:pass@host:3306/db") as cp:
    agent = Agent(..., checkpointer=cp, thread_id="user-42")
```

Constructor: `MySQLCheckpointer(dsn, *, min_pool_size=1,
max_pool_size=10)`. The DSN is a `mysql://user:pass@host:port/db` URL,
parsed into `aiomysql.create_pool` kwargs. Schema verification happens on
`__aenter__`; it never creates tables.

## What the public API exposes

The package re-exports the symbols Alembic wiring needs:

```python
from cubepi.checkpointer.mysql import cubepi_metadata, EXPECTED_SCHEMA_VERSION
```

| Symbol | Use |
|---|---|
| `cubepi_metadata` | SQLAlchemy `MetaData` — set as Alembic `target_metadata` |
| `EXPECTED_SCHEMA_VERSION` | The version cubepi verifies on connect (currently `3`) |

The SQL helpers live in the `alembic_helpers` submodule:

```python
from cubepi.checkpointer.mysql.alembic_helpers import (
    messages_partition_clause,      # the PARTITION BY KEY (...) clause for cubepi_messages
    add_pending_request_column_op,  # v1→v2 column add
    write_schema_version_op,        # returns two ';'-separated statements
)
```

## Tables

`cubepi_metadata` defines three tables:

| Table | Purpose |
|---|---|
| `cubepi_threads` | One row per thread; holds `pending_request JSON NULL` (HITL, v2+) |
| `cubepi_messages` | Append-only message log — `(thread_id, seq, role, metadata, payload, created_at)`, `PARTITION BY KEY (thread_id)` into 64 partitions |
| `cubepi_schema_version` | Single-row version marker verified on connect |

MySQL declares all partitions inline in the `CREATE TABLE` (unlike
Postgres, which uses one child table per partition). The `cubepi_messages`
table has **no foreign key** to `cubepi_threads` — MySQL forbids FKs on
partitioned tables.

## Wiring Alembic (`env.py`)

```python
# alembic/env.py
from cubepi.checkpointer.mysql import cubepi_metadata

target_metadata = cubepi_metadata
# or, alongside your own models:
# target_metadata = [your_metadata, cubepi_metadata]
```

## Autogenerate, and what to add by hand

```bash
alembic revision --autogenerate -m "add cubepi checkpointer"
```

Autogenerate emits `op.create_table(...)` for all three tables **with the
correct columns** (from `cubepi_metadata`). Two things to add by hand:

1. **KEY partitioning.** SQLAlchemy autogenerate cannot express MySQL
   `KEY` partitioning, so append the partition clause to the
   `cubepi_messages` table. Keep the autogenerated column list and add
   the clause via `op.execute`:

   ```python
   from cubepi.checkpointer.mysql.alembic_helpers import messages_partition_clause

   op.execute("ALTER TABLE cubepi_messages " + messages_partition_clause())
   ```

   `KEY` (not `HASH`) is required because `thread_id` is a `VARCHAR` —
   MySQL `HASH` partitioning only accepts integer expressions.

2. **The version row.** `write_schema_version_op()` returns two
   `;`-separated statements (DELETE then INSERT); aiomysql/pymysql run one
   statement per call, so split before executing:

   ```python
   from cubepi.checkpointer.mysql.alembic_helpers import write_schema_version_op

   for stmt in write_schema_version_op().split(";"):
       if stmt.strip():
           op.execute(stmt)
   ```

> Do not hand-trim the `cubepi_messages` columns. The checkpointer writes
> and reads `(thread_id, seq, role, metadata, payload)`; dropping any of
> them causes runtime SQL errors on the first append.

## Upgrading schema versions

cubepi bumps `EXPECTED_SCHEMA_VERSION` when the schema changes; the
checkpointer refuses to connect if the row in `cubepi_schema_version`
doesn't match. Each bump is one host migration.

**v1 → v2** (HITL): adds `pending_request JSON NULL` to `cubepi_threads`.

```python
from cubepi.checkpointer.mysql.alembic_helpers import (
    add_pending_request_column_op,
    write_schema_version_op,
)

def upgrade():
    op.execute(add_pending_request_column_op())   # ALTER TABLE ... ADD COLUMN
    for stmt in write_schema_version_op().split(";"):
        if stmt.strip():
            op.execute(stmt)
```

> MySQL does not support `ADD COLUMN IF NOT EXISTS`; guard the `ALTER`
> with an `information_schema.columns` check if the migration must be
> re-runnable.
