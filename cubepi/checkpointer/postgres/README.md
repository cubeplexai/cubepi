# Postgres checkpointer — host integration

`PostgresCheckpointer` (asyncpg) persists agent state for multi-replica,
high-concurrency deployments. **It never issues DDL at runtime** — your
application owns the schema lifecycle through Alembic, and cubepi only
reads and writes rows. Entering the async context manager
(`async with PostgresCheckpointer(...)`) verifies the schema and raises
`CubepiSchemaUninitialized` / `CubepiSchemaMismatch` if the expected
migrations haven't been applied.

Install with the extra:

```bash
pip install 'cubepi[postgres]'
```

## Using the checkpointer

```python
from cubepi.checkpointer.postgres import PostgresCheckpointer

# Entering the context manager connects the pool and verifies the schema.
async with PostgresCheckpointer("postgresql://user:pass@host/db") as cp:
    agent = Agent(..., checkpointer=cp, thread_id="user-42")
```

Constructor: `PostgresCheckpointer(dsn, *, min_pool_size=1,
max_pool_size=10)`. The DSN is whatever `asyncpg.create_pool` accepts.
Schema verification happens on `__aenter__`; it never creates tables.

## What the public API exposes

The package re-exports the symbols Alembic wiring needs:

```python
from cubepi.checkpointer.postgres import cubepi_metadata, EXPECTED_SCHEMA_VERSION
```

| Symbol | Use |
|---|---|
| `cubepi_metadata` | SQLAlchemy `MetaData` — set as Alembic `target_metadata` |
| `EXPECTED_SCHEMA_VERSION` | The version cubepi verifies on connect (currently `2`) |

The SQL helpers live in the `alembic_helpers` submodule:

```python
from cubepi.checkpointer.postgres.alembic_helpers import (
    create_message_partitions_op,   # DDL for the 64 hash partitions of cubepi_messages
    add_pending_request_column_op,  # v1→v2 column add (idempotent, IF NOT EXISTS)
    write_schema_version_op,        # sets cubepi_schema_version to EXPECTED_SCHEMA_VERSION
)
```

## Tables

`cubepi_metadata` defines three tables:

| Table | Purpose |
|---|---|
| `cubepi_threads` | One row per thread; holds `pending_request JSONB NULL` (HITL, v2+) |
| `cubepi_messages` | Append-only message log — `(thread_id, seq, role, metadata, payload, created_at)`, **HASH-partitioned** on `thread_id` into 64 partitions |
| `cubepi_schema_version` | Single-row version marker verified on connect |

## Wiring Alembic (`env.py`)

Point Alembic at cubepi's metadata so autogenerate sees the tables:

```python
# alembic/env.py
from cubepi.checkpointer.postgres import cubepi_metadata

target_metadata = cubepi_metadata
# or, alongside your own models:
# target_metadata = [your_metadata, cubepi_metadata]
```

## Autogenerate, and what to add by hand

```bash
alembic revision --autogenerate -m "add cubepi checkpointer"
```

Autogenerate emits `op.create_table(...)` for all three tables **with the
correct columns** (they come straight from `cubepi_metadata`). Two things
SQLAlchemy autogenerate does not reliably produce, so add them to the
generated migration yourself:

1. **Child partitions.** `cubepi_messages` is HASH-partitioned on
   `thread_id` (the model declares this via
   `__table_args__["postgresql_partition_by"]`). Keep the autogenerated
   `create_table` for `cubepi_messages` — including all its columns — and
   make sure it carries `postgresql_partition_by="HASH (thread_id)"`. The
   64 child partitions are not autogenerated; create them with the helper.

2. **The version row.** Autogenerate creates the `cubepi_schema_version`
   table but never inserts a row.

```python
from cubepi.checkpointer.postgres.alembic_helpers import (
    create_message_partitions_op,
    write_schema_version_op,
)

def upgrade():
    # ... autogenerated create_table() for cubepi_threads,
    #     cubepi_schema_version, and cubepi_messages (keep ALL columns;
    #     ensure cubepi_messages has postgresql_partition_by="HASH (thread_id)") ...
    op.execute(create_message_partitions_op())  # the 64 child partitions
    op.execute(write_schema_version_op())        # record EXPECTED_SCHEMA_VERSION
```

> Do not hand-trim the `cubepi_messages` columns. The checkpointer writes
> and reads `(thread_id, seq, role, metadata, payload)`; dropping any of
> them causes runtime SQL errors on the first append.

## Upgrading schema versions

cubepi bumps `EXPECTED_SCHEMA_VERSION` when the schema changes; the
checkpointer refuses to connect if the row in `cubepi_schema_version`
doesn't match. Each bump is one host migration.

**v1 → v2** (HITL): adds `pending_request JSONB NULL` to `cubepi_threads`.

```python
from cubepi.checkpointer.postgres.alembic_helpers import (
    add_pending_request_column_op,
    write_schema_version_op,
)

def upgrade():
    op.execute(add_pending_request_column_op())   # ADD COLUMN IF NOT EXISTS
    op.execute(write_schema_version_op())         # bump to EXPECTED_SCHEMA_VERSION

def downgrade():
    op.execute("ALTER TABLE cubepi_threads DROP COLUMN IF EXISTS pending_request")
    op.execute("DELETE FROM cubepi_schema_version")
    op.execute("INSERT INTO cubepi_schema_version (version) VALUES (1)")
```

`write_schema_version_op()` clears stale rows before inserting, so running
it after every cubepi upgrade keeps `cubepi_schema_version` single-valued.
